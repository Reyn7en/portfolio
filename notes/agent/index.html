Agent 笔记

基于 deeplearning.ai 公开课整理的 LangGraph 学习笔记，覆盖 State 管理、Tool Calling、ReAct 循环、记忆系统、Human-in-the-Loop、Multi-Agent 与 RAG 集成。
注：以下内容在 AI 辅助下整理。核心概念的理解、代码示例和实践总结经过了人工校对。部分 API 行为可能随 LangGraph 版本更新有变化，请以官网文档为准。

第一章 LangGraph 基础概念

State、Node、Edge、Tool Calling、Checkpointer 与 Streaming

1. 为什么学 LangGraph

在接触 LangGraph 之前，我对 Agent 的理解就是简单的 LLM 调用：扔一个 prompt，拿回结果，完事。这在单轮问答场景下没问题，但一旦涉及多轮对话、工具调用、条件分支，这种简单模式就撑不住了。

接触 LangGraph 后，最大的感受是：它把 Agent 的工作流变得显式可读了。Agent 不只是"调 LLM + 解析输出"，而是一个多步骤的工作流——每一步有明确的输入输出，步骤之间有条件跳转，中间状态需要被保存和追踪。

对我来说，LangGraph 解决了一个核心问题：如何让 Agent 的行为可控、可追踪、可中断。这在涉及工具调用和多步推理的场景中尤为关键。

2. 我对 LangGraph 的整体理解

我目前的理解是：LangGraph 是一个把 Agent 应用组织成"带状态的图"的框架。三个概念就能概括：

  • State（状态）：在所有节点间共享的数据字典，承载消息历史、工具结果和中间推理步骤。
  • Node（节点）：一个处理单元。比如"调用 LLM"是一个节点，"执行工具"是另一个节点。
  • Edge（边）：节点之间的跳转规则。普通边是固定跳转，条件边则根据当前 State 决定下一步去哪。

打个比方：如果 Agent 是一个"自动化办事流程"，State 就是那张流转中的表格，Node 是各个办事窗口，Edge 就是连接窗口的箭头。这个结构让复杂流程变得可管理。

3. State：Agent 的共享工作台

State 不是普通的 Python 变量。它最关键的特性是：每个节点返回的部分状态会和全局 State 合并（merge），而不是覆盖。不同节点更新不同字段，互不干扰。

在多轮对话中，State 通常包含：

  • messages：完整的对话历史（用户消息、AI 回复、工具调用、工具结果）
  • next_step：供条件边判断路由方向
  • 自定义字段，比如 context、intermediate_steps

LangGraph 中使用 TypedDict 或 Pydantic 模型定义 State。我目前用 TypedDict + Annotated，因为它可以为每个字段指定 reducer 逻辑（比如消息字段用 add_messages 来累积而不是覆盖）。

提醒：add_messages 的具体行为（如同 ID 消息的更新逻辑）发布前需要对照最新 LangGraph 文档核实。

4. Node 和 Edge：把流程拆成可控的步骤

在用 LangGraph 之前，我写 Agent 逻辑的方式是一个大函数吃到底：调 LLM → 解析 → 需要就调工具 → 再解析 → 返回。简单场景能跑，但问题很明显：

  • ✕ 所有逻辑搅在一起，加一个步骤就要改主函数
  • ✕ 出问题时很难判断是"LLM 判断错了"还是"工具执行错了"
  • ✕ 没有复用性，"调工具"这段逻辑每次都要重写

LangGraph 的 Node + Edge 模式解决了这个问题。每个节点是一个单一职责的函数：

# LLM Node def call_model(state: AgentState) -> dict: response = llm.bind_tools(tools).invoke(state["messages"]) return {"messages": [response]} # Tool Node def call_tools(state: AgentState) -> dict: last_message = state["messages"][-1] tool_results = [] for tool_call in last_message.tool_calls: result = execute_tool(tool_call) tool_results.append(result) return {"messages": tool_results}

条件边负责路由决策：

def should_continue(state: AgentState) -> str: last_message = state["messages"][-1] if last_message.tool_calls: return "tools" return "__end__" graph.add_conditional_edges("agent", should_continue, { "tools": "tools", "__end__": "__end__" })

5. Tool Calling：让模型从「生成文本」变成「调用能力」

这是学 LangGraph 最让我开眼界的部分。LLM 只能输出文本，但 Tool Calling 把它从"回答者"变成了"编排者"——它不执行，而是判断需要什么、然后请求执行。

完整的工具调用流程：

1LLM 接收用户输入，判断是否需要调用工具
2如果需要，LLM 不输出文本，而是输出 tool_calls（包含工具名和参数）
3条件边检测到 tool_calls，路由到工具执行节点
4工具节点实际执行工具，将结果以 ToolMessage 形式追加
5流程回到 LLM 节点，LLM 结合工具结果生成最终回答

关键认知：模型不执行工具——它请求执行。实际的工具执行发生在程序代码中。这把"决策"和"执行"分开了，意味着工具可以是一切：数据库查询、API 调用、脚本执行，甚至启动另一个 Agent。

6. Checkpointer：为什么 Agent 需要记住状态

一开始我很困惑：对话之间的状态怎么持久化？没有持久化，每次新的用户输入都从零开始，之前的工具结果和推理全都丢了。

Checkpointer 解决了这个问题，它像游戏的存档系统：

  • 每个 thread_id 对应一个独立的对话线程
  • 每经过一个图步骤，State 自动存档（checkpoint）
  • 下次用同一个 thread_id 调用，从上次存档点恢复

目前我开发调试只用内存 MemorySaver。如果要上线，需要换成持久化后端（SqliteSaver 或 PostgresSaver）。

7. Streaming：为什么流式输出很重要

graph.stream() 是我觉得最实用的功能之一。除了渐进式输出带来的用户体验提升，它对我更大的价值是调试可见性。

通过流式输出，你能看到：

  • Agent 每一步执行了哪个 Node
  • LLM 输出了什么（包括 tool_calls）
  • 工具调用的参数和返回值
  • 条件边的路由决策

没有流式输出时，我靠 print 语句追踪流程。有了流式，每一步都清清楚楚，调试效率大幅提升。把这个过程暴露给前端，也能让用户看到 Agent 的"思考过程"，增加信任感。

8. 一个最小工作流例子

一个最小的 LangGraph 工作流，展示了 ReAct 循环的核心逻辑：

from typing import TypedDict, Annotated, Sequence from langgraph.graph import StateGraph, END from langgraph.graph.message import add_messages from langgraph.checkpoint.memory import MemorySaver class AgentState(TypedDict): messages: Annotated[Sequence, add_messages] def agent_node(state: AgentState): response = model_with_tools.invoke(state["messages"]) return {"messages": [response]} def tool_node(state: AgentState): last_msg = state["messages"][-1] results = [execute(tc) for tc in last_msg.tool_calls] return {"messages": results} def router(state: AgentState): last_msg = state["messages"][-1] if getattr(last_msg, "tool_calls", None): return "tools" return END builder = StateGraph(AgentState) builder.add_node("agent", agent_node) builder.add_node("tools", tool_node) builder.set_entry_point("agent") builder.add_conditional_edges("agent", router) builder.add_edge("tools", "agent") graph = builder.compile(checkpointer=MemorySaver()) config = {"configurable": {"thread_id": "1"}} events = graph.stream( {"messages": [HumanMessage(content="weather?")]}, config ) for event in events: print(event)

如果 LLM 判断需要天气工具，流程是：agent → tools → agent → END。如果不需要：agent → END。整个过程逐步流式输出，Agent 的决策链一目了然。

9. 我的阶段性理解

  • 它不是聊天封装，是工作流引擎。LangGraph 把 Agent 的每一步（调用模型、执行工具、条件跳转）显式组织成图，流程可控可追踪。
  • State 是核心。理解了 State 的 merge 机制和 reducer 设计，基本就理解了 LangGraph 的一半。
  • 图和条件边让流程显式化。Agent 不再是黑盒——每一步、每个分支都可追踪。
  • Tool Calling 不神秘。它就是一个结构化的"请求-执行-反馈"循环，LLM 只做判断。
  • 我目前处于"能用"阶段，离"用好"还有距离。复杂路由场景、异常处理、多工具时的性能——这些都还在学习中。
第二章 State 深入

add_messages、Reducer 与 StateGraph 选型

1. 踩坑实录：重新理解 State

在第一篇笔记里，我把 State 概括为「Agent 的共享工作台」。这个说法够用，但真正写代码的时候会发现：State 怎么定义、怎么更新、多个节点怎么安全地读写，这些细节直接决定图能不能跑通。

我一开始的理解是：「State 就是一个字典，节点往里面塞东西就行」。后来踩了几次坑才发现，LangGraph 的 State 设计有一套明确的规则，不理解这些规则，图的行为会和你预期的不一样。

2. TypedDict + Annotated：State 的标准写法

LangGraph 推荐的 State 定义方式是 TypedDict + Annotated。起初不太理解为什么要搞这么复杂，直接用 dict 不行吗？

后来意识到，LangGraph 需要回答两个问题：

  • State 里有哪些字段？（类型检查、IDE 补全）
  • 每个字段怎么合并？（多个节点写入同一字段时的冲突处理）
from typing import TypedDict, Annotated from langgraph.graph import StateGraph from langchain_core.messages import add_messages class AgentState(TypedDict): messages: Annotated[list, add_messages] step: int

Annotated 的第二个参数 add_messages 就是「合并函数」（reducer）。当多个节点都往 messages 写值时，LangGraph 不是直接覆盖，而是用 add_messages 把新消息追加到列表里。

我目前只用过 add_messages 这一种 reducer。官方文档里还有其他用法（比如 operator.add 用于数值累加），这些我还没实际用过，后续学到了再补充。

3. add_messages 到底在做什么

这是我最开始困惑的地方。add_messages 听起来像「把消息追加到列表末尾」，但实际上它的行为更智能：

  • 如果新消息的 id 和已有消息相同 → 覆盖（更新）
  • 如果 id 不同 → 追加

这意味着：如果 LLM 生成了一条带 tool_calls 的助手消息，随后工具执行结果回来，add_messages 会把工具结果追加，而不是覆盖之前的助手消息。这正是多轮对话需要的行为。

我目前的理解是：add_messages 的设计目标是「让消息列表始终是一棵有序的对话树，而不是一堆乱序的碎片」。这个理解对不对，我还在验证。

4. Reducer 函数：多个节点写入同一字段怎么办

如果 State 里有一个 step 字段，节点 A 写 step=1，节点 B 写 step=2，LangGraph 用哪个值？

答案是：取决于这个字段有没有声明 reducer。

  • 有 reducer（如 add_messages）→ 按 reducer 逻辑合并
  • 没有 reducer → 后执行的节点覆盖先执行的节点（最后一次写入生效）

我之前踩过一个坑：State 里定义了一个 results: list，但忘记加 Annotated[..., operator.add]。结果每次节点写入 results，之前的就被覆盖了。加上 reducer 之后才正常累加。

这个坑让我意识到：LangGraph 不会自动帮你「合并」任何东西，除非你显式声明 reducer。

5. MessageGraph vs StateGraph：该用哪个？

LangGraph 还提供了 MessageGraph，这是一个专门针对「消息列表」场景的简化版 StateGraph。两者的区别：

  • MessageGraph：State 只有一个 messages 字段，适合纯对话场景，写法更简洁
  • StateGraph：State 可以有任意多个字段，适合需要管理多种状态（消息、步骤计数、检索结果、工具输出等）的场景

我目前所有的练习都用的是 StateGraph，即使场景很简单。原因是：我担心用 MessageGraph 之后，如果后面想加一个新字段（比如 search_results），又要迁移到 StateGraph，不如一开始就直接用更通用的方案。当然，如果场景确实只有一个 messages 列表且确定不会扩展，MessageGraph 更简洁——但我的习惯是直接上 StateGraph。

6. 实际写代码时的几个观察

以下是我在实际写代码过程中逐渐发现的几点：

  • 节点函数的签名是固定的。每个节点函数接收当前 State，返回一个「部分 State」字典。这个设计一开始我觉得有点怪（为什么不是直接修改 State？），后来理解：返回部分 State + reducer 合并，避免了多个节点同时修改 State 带来的竞态问题。
  • State 的「当前值」在节点执行期间是不变的。节点 A 读取到的 State，是图开始执行这个节点那一刻的快照。如果节点 B 和 A 并行执行，它们看到的 State 是同一个快照，互相不会影响。这个特性我还没有在实际项目里深度用到，但目前的理解是这样的。
  • 调试时打印 State 很有用。在每个节点函数开头加一句 print(state)，可以看到图执行到这一步时的完整状态。这比单纯看 LangSmith 的 trace 更直观（对我而言）。

7. 本章回顾

  • State 不是普通变量，它是图的「共享工作台」，所有节点通过它通信。
  • TypedDict + Annotated 是标准写法，后者声明 reducer，决定字段怎么合并。
  • add_messages 不是简单追加，而是按消息 id 智能合并（覆盖或追加）。
  • 没声明 reducer 的字段，最后一次写入生效（覆盖）。这个行为要特别小心。
  • MessageGraph 是 StateGraph 的特化版本，适合纯对话场景，但我还没对比过两者的实际差异。
第三章 Router & Tool Calling Agent

条件边路由、结构化输出、ToolNode 与 bind_tools

1. 让 Agent 选路而不是硬编码

最开始的 Agent 只有一条路：用户输入 → LLM → 输出。但实际场景下，不同问题需要不同的处理方式。比如「天气查询」和「写代码」，显然应该走不同的工具链。

Router Agent 的核心思路是：让 LLM 先判断「该走哪条路」，再执行对应的流程。这比把所有工具都塞给 LLM 让它自己选要清晰得多。

我目前的理解：Router 本质上是一个「分类器」——输入问题，输出一个路由标签，图根据这个标签走到不同的下游节点。

2. 条件边：LangGraph 的路由机制

在 LangGraph 里，路由不是靠 if/else 写在代码里，而是靠条件边（conditional edge）声明。

from langgraph.graph import StateGraph, END def router_function(state: AgentState) -> str: # 根据 state 内容决定下一个节点 if state["messages"][-1].content == "天气": return "weather_tool" else: return "chat_model" # 添加条件边 graph.add_conditional_edges( "router", # 来源节点 router_function, # 路由函数（返回下一个节点名） { "weather_tool": "weather_tool", "chat_model": "chat_model", END: END, } )

关键点：router_function 接收当前 State，返回一个字符串（下一个节点的名称）。LangGraph 根据这个返回值决定图的下一步走向。

我目前只写过简单的 router_function（基于关键词匹配）。也可以让 LLM 来生成这个路由判断，这个我还没实践过，后续学到 ReAct 部分会深入。

3. 结构化输出：让路由判断更可靠

如果让 LLM 自由文本输出「该走哪条路」，解析起来很不稳定。更好的方式是：强制 LLM 输出结构化结果（比如 JSON，或者 Pydantic 模型）。

LangChain 提供了几种方式：

  • with_structured_output() — 推荐，直接绑定 Pydantic 模型
  • 提示工程 + 解析器 — 手动解析 LLM 输出，不够稳定
  • JSON Schema — 比 Pydantic 更底层，不太直观
from pydantic import BaseModel, Field class RouterOutput(BaseModel): next_step: str = Field(..., description="下一步：weather / chat / END") # 绑定到模型 model_with_structure = llm.with_structured_output(RouterOutput) # 在 router_function 里调用 def router_function(state): response = model_with_structure.invoke(state["messages"]) return response.next_step # 直接返回字符串，用作节点名

这样做的好处：LLM 的输出被约束为固定格式，解析简单且稳定。我目前只试过 with_structured_output() 的简单用法，更复杂的嵌套结构还没实践过。

4. Tool Calling Agent：从「生成文本」到「调用能力」

Tool Calling 的本质是：LLM 不直接生成最终回答，而是生成「工具调用请求」，由程序执行工具后把结果放回 State，再让 LLM 继续。

标准的 Tool Calling 流程：

  1. 用户输入 → LLM 判断是否需要工具
  2. 如果需要 → LLM 输出 tool_calls（工具名 + 参数）
  3. 程序执行工具，拿到结果
  4. 工具结果追加到 messages 列表
  5. LLM 再次被调用，基于完整上下文生成最终回答

我目前的理解：Tool Calling 不是一个「魔法」，它就是「LLM 生成结构化输出 → 程序执行 → 结果回到 LLM」的循环。理解了这个循环，后面的 ToolNode、bind_tools() 都好理解。

5. ToolNode：工具执行的专用节点

手动写「解析 LLM 的 tool_calls → 执行工具 → 把结果写回 State」这个流程很繁琐。LangGraph 提供了 ToolNode 来封装这个逻辑。

from langgraph.prebuilt import ToolNode # 定义工具列表 tools = [get_weather, search_web, calculate] # 创建 ToolNode（自动执行 tools 列表里的工具） tool_node = ToolNode(tools) # 在图里添加 graph.add_node("tools", tool_node) graph.add_conditional_edges("agent", should_continue) graph.add_edge("tools", "agent") # 工具执行完，回到 agent

ToolNode 做了什么：读取 State 里最后一个消息的 tool_calls，逐个执行对应的工具，把每个工具的结果作为 ToolMessage 追加到 messages。

我目前只用过 ToolNode 的默认行为。官方文档里提到可以自定义工具执行逻辑（比如加重试、加权限检查），这些我还没用到。

6. bind_tools()：把工具告诉 LLM

LLM 本身不知道有哪些工具可用。bind_tools() 的作用就是：把工具的定义（名称、描述、参数 Schema）注入到 LLM 的调用上下文里，这样 LLM 才知道「我可以调用这些工具」。

from langchain_openai import ChatOpenAI from langchain_core.tools import tool @tool def get_weather(location: str) -> str: """获取指定地点的天气。""" return f"{location} 的天气是：晴天，25°C" llm = ChatOpenAI(model="gpt-4o") llm_with_tools = llm.bind_tools([get_weather]) # 现在 LLM 知道 get_weather 这个工具 response = llm_with_tools.invoke(messages) # response.tool_calls 里会有工具调用请求

关键点：bind_tools() 返回的是一个新的 LLM 对象（原对象不变）。这个设计我一开始有点不习惯，后来理解：这样可以在不同的节点里绑定不同的工具列表。

7. 一个完整的 Tool Calling Agent 骨架

把上面几节串起来，一个最简骨架：

from typing import Annotated from typing_extensions import TypedDict from langgraph.graph import StateGraph, END from langgraph.prebuilt import ToolNode from langchain_openai import ChatOpenAI from langchain_core.messages import add_messages class AgentState(TypedDict): messages: Annotated[list, add_messages] tools = [...] # 工具列表 llm = ChatOpenAI().bind_tools(tools) def agent_node(state: AgentState): response = llm.invoke(state["messages"]) return {"messages": [response]} def should_continue(state: AgentState) -> str: last_msg = state["messages"][-1] if hasattr(last_msg, "tool_calls") and last_msg.tool_calls: return "tools" return END graph = StateGraph(AgentState) graph.add_node("agent", agent_node) graph.add_node("tools", ToolNode(tools)) graph.add_conditional_edges("agent", should_continue) graph.add_edge("tools", "agent") graph.set_entry_point("agent")

8. 本章回顾

  • Router 是分类器。条件边 + 路由函数，决定图的下一步走向。
  • 结构化输出让路由更稳定。with_structured_output() + Pydantic，避免自由文本解析的不稳定性。
  • Tool Calling 是循环。LLM 生成 tool_calls → 程序执行 → 结果回 State → LLM 继续。
  • ToolNode 封装了工具执行逻辑。不需要手动解析 tool_calls、手动调用工具。
  • bind_tools() 告诉 LLM 有哪些工具。返回新 LLM 对象，原对象不受影响。
第四章 ReAct 自治循环

Thought→Action→Observation、create_react_agent、流式输出

1. 让 Agent 自己决定下一步

之前的 Router Agent 和 Tool Calling Agent，流程都是「我（开发者）定好路线，Agent 按路线走」。但真实场景里，有些任务的步骤数、决策路径是事先没法写死的。

比如「帮我查北京天气，如果下雨就推荐室内活动」。这需要：查天气 → 判断结果 → 决定下一步。这个「判断→执行→再判断」的循环，就是 ReAct。

我目前的理解：ReAct Agent 的核心就是让 LLM 自己决定「下一步做什么」，而不是把所有分支都写死在代码里。

2. ReAct 循环：Thought → Action → Observation

ReAct 的全称是 Reasoning + Acting，循环步骤：

  1. Thought：LLM 思考当前状态和目标，决定下一步
  2. Action：LLM 输出工具调用（tool_call）
  3. Observation：工具执行结果回到 State
  4. 回到 1，直到 LLM 判断可以输出最终答案
# ReAct 循环伪代码 while not done: response = llm.invoke(state["messages"]) if response.has_tool_calls: for tool_call in response.tool_calls: result = execute_tool(tool_call) state["messages"].append(result) else: break # LLM 输出了最终答案

我一开始不理解「为什么 LLM 知道什么时候停」。后来明白：靠的是训练时的指令遵循能力——当不需要再调用工具时，LLM 直接生成普通文本回复，而不是 tool_calls。

3. create_react_agent：一行搭建 ReAct Agent

LangGraph 提供了 create_react_agent 这个预置函数，一行代码就能搭建一个完整的 ReAct Agent。

from langgraph.prebuilt import create_react_agent agent = create_react_agent( model=ChatOpenAI(model="gpt-4o"), tools=[get_weather, search_web, calculate], state_modifier="你是一个有帮助的助手，使用工具来回答用户问题。" ) # 使用：直接 invoke result = agent.invoke({"messages": [HumanMessage(content="北京天气怎么样？")]})

这行代码背后做了什么：创建了一个 StateGraph，包含 agent 节点（调用 LLM）、tools 节点（执行工具）、条件边（判断是否有 tool_calls，决定下一步），以及循环逻辑。

我目前只用过 create_react_agent 的默认行为。官方文档里提到可以自定义 state_modifier（相当于系统提示）、自定义中断条件等，这些我还没实践过。

4. 流式输出：看到 Agent 的思考过程

ReAct Agent 的循环可能有很多步，如果不流式输出，用户只能看到「thinking...」然后突然收到一大段答案。体验很差。

LangGraph 提供了几种流式模式：

  • stream_mode="values"：每次 State 更新后，输出完整 State
  • stream_mode="updates"：只输出这次更新了 State 的哪部分（更精简）
  • stream_mode="messages"：逐个 token 输出 LLM 的回复（最像聊天体验）
from langgraph.prebuilt import create_react_agent agent = create_react_agent(...) # 流式调用 for chunk in agent.stream( {"messages": [HumanMessage(content="北京天气，如果下雨推荐室内活动")]}, stream_mode="messages" ): # chunk 是 (message, metadata) 元组 print(chunk[0].content, end="", flush=True)

我目前的体会：stream_mode="messages" 最适合和用户交互的场景。"values" 更适合调试，因为你能看到每一步完整的 State。

5. 终止条件：Agent 什么时候停

ReAct Agent 的终止条件很朴素：LLM 不再输出 tool_calls，而是输出普通文本。此时 create_react_agent 内部的循环就结束了。

但有些边界情况需要小心：

  • LLM 陷入死循环（一直输出 tool_calls，但每次结果都一样）→ 需要设最大步数限制
  • 工具执行失败，LLM 不知道怎么处理 → 需要在 State 里加入错误信息，让 LLM 有机会调整
  • LLM 「觉得」任务完成了，但实际没完成 → 需要在系统提示里强调完成标准
我目前没有在实际项目里遇到死循环问题。LangGraph 提供了 max_iterations 参数来设上限，但我还没用过。后续实践后再补充。

6. ReAct Agent vs Tool Calling Agent：什么时候用哪个

我目前的判断标准：

  • 流程固定、步骤明确 → Tool Calling Agent（用条件边控制流程）
  • 步骤数不固定、需要 LLM 自主决策 → ReAct Agent（让 LLM 自己判断下一步）

举个例子：「用户问天气 → 调用天气工具 → 回复」这个流程是固定的，用 Tool Calling Agent 就够了。「帮我研究一下 XX 领域的最新进展」这种开放任务，步骤数没法事先确定，适合用 ReAct。

当然，真实场景中两者的边界经常是模糊的。比如一个看似固定的流程里夹杂了需要自主判断的步骤——这时与其硬要分类，不如直接用 ReAct 模式兜底，灵活性更高。

7. 本章回顾

  • ReAct = Reasoning + Acting。LLM 自己决定下一步，而不是开发者写死所有分支。
  • 循环终止条件很简单。LLM 不输出 tool_calls 了，就停了。
  • create_react_agent 一行搭建，但背后是一整套 StateGraph + 条件边 + 循环逻辑。
  • 流式输出很重要。ReAct 可能有很多步，不流式的话用户体验很差。
  • 和 Tool Calling Agent 的选择取决于流程是否固定——复杂场景推荐直接用 ReAct 兜底。
第五章 记忆系统

Checkpointer 短期记忆、Store 长期记忆、混合使用

1. 记忆的两个维度：会话内 vs 跨会话

在第一篇笔记里，我提到 LangGraph 的图每次执行都是「无状态」的：你调 graph.invoke(state)，它跑完就结束了，下次再调，它不记得上次发生了什么。

这个问题在两类场景下会变得很突出：

  • 多轮对话：用户说「帮我查一下北京明天的天气」，然后接着说「那上海呢？」——Agent 需要知道上下文是「查天气」。
  • 长时任务：一个任务需要分多次执行（比如今天做一半，明天继续），Agent 需要记住之前做到了哪一步。

LangGraph 把「记忆」拆成了两层：Checkpointer（短期记忆）和Store（长期记忆）。我一开始没搞清楚这两者的区别，用错了好几次。

2. Checkpointer：短期记忆（会话级别）

Checkpointer 的作用是：让图在多次调用之间，记住上一次执行的完整 State。它本质上是把「图的执行快照」存到一个地方，下次调同一个线程时，自动恢复。

  • Checkpointer 是按「线程」（thread）组织的。同一个 thread_id，多次调用 graph.invoke() 或 graph.stream()，State 会延续。
  • 不同 thread_id 之间是隔离的，互不影响。
  • Checkpointer 存的是「完整的 State 快照」，不是差分。

我目前的用法是：给每个用户会话分配一个唯一的 thread_id，然后把这个 thread_id 传给 graph.invoke() 的 config 参数。这样即使用户关掉页面再打开，只要 thread_id 不变，对话历史就在。

我目前只用过 MemorySaver（内存存储，重启即丢）和 SqliteSaver（持久化到 SQLite）。生产环境应该用持久化的 Checkpointer，否则重启后所有会话都没了。

3. Store：长期记忆（跨会话）

Checkpointer 解决的是「同一次对话（同一个 thread）里，图怎么记住之前的状态」。但有些信息是需要跨会话保留的，比如：

  • 用户偏好（「我喜欢简洁的回答」）
  • 用户画像（「这个用户是研究生，有编程基础」）
  • 长期积累的知识（「这个用户之前问过 RAG，我可以引用之前的讨论」）

Store 就是干这个的。它是一个键值存储（key-value store），可以按 namespace（命名空间）和 key（键）来存取数据。

Store 适合存「和用户相关但不属于当前对话 State」的信息，而 Checkpointer 适合存「当前对话的进展状态」。这里容易混淆，需要留意两者的边界。

这里有一个容易混淆的点：Store 里的数据，不会自动进入图的 State。你需要在节点里显式地去 Store 里读，然后决定怎么用。这个设计和 Checkpointer 不一样（Checkpointer 是自动恢复的）。

4. MemorySaver vs SqliteSaver vs PostgresSaver

  • MemorySaver：存在内存里，重启即丢。适合开发调试，不适合生产。
    我目前本地调试都用这个，简单够用。
  • SqliteSaver：存到 SQLite 文件，重启不丢。适合单机部署、小规模应用。
    我试过一次，配置比 MemorySaver 麻烦一点，但也不算难。
  • PostgresSaver：存到 PostgreSQL，支持分布式。适合生产环境、多实例部署。
    我还没用过，但看文档应该是标准 PostgreSQL 连接配置。

选哪个，取决于你的部署规模和对持久化的需求。我目前的建议是：开发用 MemorySaver，上线前切换到 SqliteSaver 或 PostgresSaver。

5. 如何在图中使用 Checkpointer

from langgraph.checkpoint.memory import MemorySaver checkpointer = MemorySaver() graph = graph_builder.compile(checkpointer=checkpointer) # 调用时传入 thread_id config = {"configurable": {"thread_id": "user-123"}} graph.invoke(initial_state, config) # 下次再用同一个 thread_id 调用，State 会自动恢复 graph.invoke(new_input, config)

我踩过的一个坑是：

  • thread_id 是在 config 的 configurable 字典里传的，不是直接传 thread_id=xxx。
  • 如果忘记传 config，Checkpointer 不会报错，但它也不会恢复 State——相当于每次都是全新的图。

另一个观察：Checkpointer 是按「图」级别的。如果你有两个图，它们需要共享记忆，要么用同一个 Checkpointer 实例，要么用 Store（Store 是跨图的）。

6. 如何在图中使用 Store

Store 在图的编译阶段传入，节点通过 config 访问。以下代码基于 LangGraph 文档的 API，在基础场景下可以直接使用：

  • 在编译图的时候，传入一个 store 实例（类似 Checkpointer 的传法）。
  • 在节点函数里，通过 config 拿到 store，然后调用 store.get(namespace, key) 或 store.put(namespace, key, value)。
  • Store 的数据不会自动进入 State，需要手动读取、手动决定怎么用。
from langgraph.store.memory import InMemoryStore store = InMemoryStore() graph = graph_builder.compile(store=store) # 在节点里读写 Store def my_node(state, config): store = config["store"] namespace = ("user_profiles", config["configurable"]["user_id"]) # 读 profile = store.get(namespace, "prefs") # 返回 dict 或 None # 写 store.put(namespace, "prefs", {"language": "zh", "detail": "high"}) # 用读取到的数据影响 State if profile and profile.get("language") == "zh": return {"system_prompt": "请用中文回答"} return {"system_prompt": "Please answer in English"}

7. 混合使用场景

我目前能想象到的，Checkpointer 和 Store 混合使用的典型场景是：

  • 多轮对话 Agent：Checkpointer 记住当前对话的 State（消息历史、工具调用结果），Store 记住用户偏好（「这个用户喜欢详细的解释」）。
  • 长期任务 Agent：Checkpointer 记住任务执行到了哪一步（「正在执行第 3 个工具」），Store 记住任务相关的背景知识（「这个任务涉及的数据源是 XXX」）。
  • 多用户 Agent 系统：每个用户有自己的 Checkpointer thread，所有用户共享一个 Store（存系统级的知识库元数据）。

我还没有在实际项目里同时用过这两者。目前的理解是：Checkpointer 管「对话进行到哪里了」，Store 管「关于这个用户/任务我知道什么」。

8. 本章回顾

  • LangGraph 的记忆系统分两层：Checkpointer（短期，会话级）和 Store（长期，跨会话）。
  • Checkpointer 按 thread_id 组织，同一个 thread 多次调用会自动恢复 State。
  • Store 是键值存储，需要手动在节点里读写，不会自动进入 State。
  • 开发用 MemorySaver，生产用 SqliteSaver 或 PostgresSaver。
  • Store 的使用场景在实际项目中不算高频——大部分需求 Checkpointer 就能覆盖。当需要跨会话共享数据（如用户偏好、知识库元数据）时再考虑引入 Store。
第六章 Human-in-the-Loop

interrupt / Command、审批流设计、敏感操作安全实践

1. 什么时候必须让用户介入

LangGraph 的核心理念之一是把 Agent 的每一步都变成显式的、可中断的、可恢复的。这和传统 Agent 框架很不一样——很多框架把 LLM 的推理过程封装成一个黑盒，你想让人审一下再执行，往往要在 prompt 层面 hack，非常不优雅。

实际场景里，有几类情况你必须让人介入：

  • 高风险操作：发邮件、转账、删除数据——执行前必须人工确认。
  • 不确定性高：模型置信度低，或者多个候选 action 得分接近，需要人做最终判断。
  • 合规要求：金融、医疗等领域，某些决策必须有审计轨迹和人工签字。
  • 长期运行的 Agent：Agent 跑了几轮后状态已经很复杂，人需要定期 review，决定继续还是调整方向。
LangGraph 的设计哲学：中断不是异常，而是正常流程的一部分。 interrupt 是图中的一个普通节点逻辑，Checkpointer 负责保存现场，Command 负责恢复执行。

2. interrupt：在节点中主动中断

interrupt() 是 LangGraph 提供的一个特殊函数，调用它时，图的执行会立即暂停，并把中断信息返回给调用方。等人工处理完，再通过 Command 恢复执行。

from langgraph.types import interrupt def human_approval_node(state: State) -> dict: # 准备一个需要人工确认的信息 draft = state["draft_reply"] # 中断！执行到这里会暂停，把 draft 返回给调用方 approval = interrupt( {"question": "请审核这封邮件，是否发送？", "draft": draft} ) # 恢复后，approval 里包含了人工的输入 return {"approved": approval["approved"], "comment": approval["comment"]}

关键点：interrupt() 接收的参数会被原封不动地返回给调用方（通常是你的后端 API 或前端）。调用方拿到这个信息后，展示给人工，收集输入，再通过 Command 恢复图。

中断的典型位置有两个：

  • interrupt_before：在某个关键节点执行前中断（比如发邮件工具调用前）。
  • interrupt_after：在某个节点执行完后、下一个节点开始前中断（比如模型生成了回复草稿，让人审核后再发）。

实际上 API 层面就是一个 interrupt() 调用，放在节点函数的合适位置就行。「before/after」是你放的位置不同，不是两个不同函数。

interrupt 并不会抛异常或修改 State。它更像是一个「协程 yield」——把控制权交还给调用方，State 保持不变，等恢复时从 interrupt 调用后的下一行继续执行。

3. Command：恢复执行与注入人工输入

中断后怎么恢复？用 Command。它是 LangGraph 提供的另一个特殊类型，用来向图中注入外部输入并指定恢复位置。

from langgraph.types import Command # 假设图的当前状态被 interrupt 暂停了 # 前端收集到人工审批结果后，调用后端 API def resume_graph(thread_id: str, approved: bool, comment: str): # Command 告诉图：从哪个节点恢复（通常是被中断的节点），以及注入什么数据 command = Command( goto="human_approval_node", # 恢复到哪个节点 update={"approved": approved, "comment": comment} # 注入到 State 的数据 ) # 用同一个 thread_id 继续运行 result = app.invoke(command, config={"configurable": {"thread_id": thread_id}}) return result

这里有一个我初学时容易混淆的点：Command 的 goto 指定的是「恢复后从哪个节点继续执行」，而 update 是直接 merge 进当前 State 的数据。恢复后，节点函数从 interrupt() 调用后的那行继续跑，同时 State 里已经有了人工输入的数据。

恢复时不需要从头重跑。Checkpointer 已经把中断前的完整 State 存好了，恢复时 LangGraph 会从断点继续，之前的节点不会重复执行。这对长链路 Agent 来说非常重要——你不会想让人审批一次就把前面 10 步全部重跑一遍。

4. 审批流设计模式

以下是一个基于「邮件发送 Agent」场景的 Human-in-the-Loop 案例，整理成一个可复用的设计模式。

模式：审批节点（Approval Node）

  1. 模型生成一个「待审批操作」（比如邮件草稿），写入 State。
  2. 下一个节点是「审批节点」，里面调用 interrupt()，把草稿内容返回给调用方。
  3. 前端/API 展示草稿，收集用户决策（批准 / 修改 / 拒绝）。
  4. 用 Command 恢复，注入审批结果。
  5. 审批节点根据结果决定下一个节点：批准 → 执行发送；拒绝 → 结束或重新规划。
from langgraph.graph import END def after_approval(state: State) -> str: if state["approved"]: return "send_email_node" # 批准 → 执行 else: return END # 拒绝 → 结束 # 在图里 graph.add_conditional_edges("human_approval_node", after_approval)

这个模式的好处是：审批逻辑是图的一个普通节点，不是特殊处理。你可以对它做所有对普通节点能做的事——加工具调用、加条件边、在前面加一个 LLM 节点来「预处理」待审批内容。

一个实用的设计：如果审批被拒绝，可以让图转到一个新的「重新规划」节点，让模型根据拒绝原因重新生成方案，而不是直接 END。这样 Agent 有自我修正的机会，体验更好。

5. 敏感操作的安全实践

学完这部分后我总结了几条实际写代码时需要注意的安全实践：

  • 永远在工具执行前中断。不要等工具执行完了再让人审核——那时候副作用已经发生了。正确做法是在调用敏感工具的节点前放一个审批节点。
  • 中断信息要足够详细。interrupt() 的参数就是你展示给用户的内容，尽量结构化（操作类型、目标、参数、预计影响），让人能做出 informed decision。
  • 恢复时验证输入。人工输入的数据（特别是自由文本）可能有毒或不合理，恢复后在节点里做一层校验再往下走。
  • 超时处理。如果人工一直不审批怎么办？需要设计超时机制（比如 10 分钟没响应自动拒绝或走备用流程）。
  • 审计日志。每次中断 + 恢复都要记录日志（谁审批的、什么决策、时间戳），方便事后审计和 debug。

推荐的做法是：把「待执行操作」和「执行结果」分开存在 State 里。中断时展示「待执行操作」，恢复后执行，执行完把结果写回 State。这样整个流程的记录是完整的，出了问题可以回溯。

6. 中断与 Checkpointer 的配合

这一部分我在学的时候觉得是 LangGraph 设计最巧妙的地方之一：interrupt 能工作，完全依赖 Checkpointer。

from langgraph.checkpoint.sqlite import SqliteSaver # 用持久化的 Checkpointer with SqliteSaver.from_conn_string("checkpoints.db") as saver: app = graph.compile(checkpointer=saver) # 第一次运行，到 interrupt 处暂停 thread_id = "email-approval-001" result = app.invoke( {"messages": [("user", "帮我回复这封邮件")]}, config={"configurable": {"thread_id": thread_id}} ) # result 里会包含 interrupt 的信息，图的状态已存入 checkpoints.db # --- 这里可以停很久，甚至重启进程 --- # 恢复：用同一个 thread_id command = Command( goto="human_approval_node", update={"approved": True, "comment": "看起来没问题"} ) result = app.invoke( command, config={"configurable": {"thread_id": thread_id}} )

这个设计让我意识到：Human-in-the-Loop 不是 LangGraph 的「附加功能」，而是和 Checkpointer 深度集成的核心能力。没有持久化，Human-in-the-Loop 就只能做同步审批（人在 loop 里等着），有了持久化，可以做异步审批（发个链接，几小时后再回来继续）。

7. 流式输出下的人工介入

以下结合官方文档整理流式输出中人工介入的实现思路。

  1. 模型正在流式输出一段文字……
  2. 你在前端实时展示这段文字，同时用另一个模型或规则检测内容是否「不安全」。
  3. 如果检测到危险内容，立即调用一个「中断接口」，让当前图的执行暂停。
  4. 前端弹出确认框：「模型正在生成可能不合适的内容，是否继续？」
  5. 用户选择「停止」→ 图走到一个「安全终止」节点；选择「继续」→ 恢复流式输出。

一个更简单的替代方案是：不在流式过程中介入，而是在模型输出完成后、工具执行前介入。这样实现简单很多，也能覆盖大多数场景。

8. 小结

  • interrupt() 是 LangGraph 实现 Human-in-the-Loop 的核心，调用时图暂停，State 保持不变。
  • Command 用来恢复执行，指定恢复节点并注入人工输入，不需要从头重跑。
  • 审批流设计：审批节点是普通节点，用条件边根据审批结果决定下一步走向。
  • 中断 + Checkpointer 配合，支持异步审批（跨进程、跨时间）。
  • 敏感操作的安全实践：执行前中断、结构化中断信息、恢复时校验输入、记录审计日志。
  • 流式输出中的实时介入实现较复杂，建议改成「输出完成后介入」更简单。
Human-in-the-Loop 的概念体系已比较清晰。如果后续实践中遇到坑，会回来补充。
第七章 Multi-Agent 架构

Supervisor、Hierarchical、平级协作、Agent 间通信

1. 单 Agent 的天花板在哪

在接触 Multi-Agent 架构之前，我写的 Agent 都是「单兵作战」：一个图，一个 LLM，一把工具。但当任务复杂度上升之后，单体 Agent 会遇到几个天花板：

  • Prompt 太长、太杂。一个 Agent 既要懂代码，又要懂法律，还要懂金融——prompt 塞不下，效果也差。
  • 工具太多，选择困难。给一个 Agent 绑 30 个工具，它经常选错或者用错参数。
  • 单点失败。一个 Agent 卡住或者进入死循环，整个流程就挂了。
  • 无法并行。单体 Agent 本质上是串行的，多个子任务无法同时推进。

Multi-Agent 的思路是：把复杂任务拆给多个专门化的 Agent，每个 Agent 只管自己擅长的事。就像软件工程里的「单一职责原则」，Agent 也该如此。

Harrison Chase 曾指出：「Agent 的边界应该和人类的角色边界对齐」。一个研究团队里有研究员、有工程师、有项目经理，Agent 团队也该这样分工。

2. Supervisor 模式

Supervisor（监督者）模式是 Multi-Agent 最直观的架构：一个中央 Agent（Supervisor）负责任务分发和结果汇总，多个 Worker Agent 负责具体执行。

from langgraph.graph import StateGraph, END from typing import Annotated, TypedDict from langgraph.graph.message import add_messages class TeamState(TypedDict): messages: Annotated[list, add_messages] next: str # 下一个要执行的 Agent 名称 task: str results: dict # 各 Agent 的执行结果 # Supervisor 节点：决定下一个派给谁 def supervisor_node(state: TeamState): # 用一个「擅长调度」的模型来决定任务分配 response = supervisor_llm.invoke([ ("system", "你是一个项目管理者，决定下一个该派给谁：coder / researcher / writer / FINISH"), ("user", f"当前任务：{state['task']}\n已有结果：{state['results']}\n请决定下一步。") ]) next_agent = response.content.strip() return {"next": next_agent} # 构建图 graph = StateGraph(TeamState) graph.add_node("supervisor", supervisor_node) graph.add_node("coder", coder_agent) # 每个 Worker 是一个编译好的 LangGraph 应用 graph.add_node("researcher", researcher_agent) graph.add_node("writer", writer_agent) # 条件边：supervisor 决定下一步去哪 graph.add_conditional_edges("supervisor", lambda s: s["next"]) graph.set_entry_point("supervisor")

关键点：每个 Worker Agent 本身可以是一个完整的 LangGraph 应用（用 as_node 或 with_types 包装成节点）。Supervisor 不用关心 Worker 内部怎么实现，只要知道它能处理什么类型的子任务。

3. Hierarchical 分层架构

Supervisor 模式有一个问题：当 Worker 数量变多，Supervisor 的 prompt 和决策压力也会变大。Hierarchical（分层）架构解决这个问题——它不是扁平的「一对多」，而是「多层金字塔」。

  1. L1 - 总控 Agent：接收用户任务，拆分成若干子目标，派给 L2 的组长 Agent。
  2. L2 - 组长 Agent：每个组长管 2-3 个执行 Agent，负责协调组内任务、汇总结果。
  3. L3 - 执行 Agent：具体干活的，绑定具体工具和 prompt。
from langgraph.graph import StateGraph # L3：执行层（具体干活） coder_graph = StateGraph(CoderState) coder_graph.add_node("write_code", write_code_node) # ... 构建 coder_graph ... # L2：组长层（协调多个 L3） backend_lead_graph = StateGraph(TeamState) backend_lead_graph.add_node("coder", coder_graph.compile()) # 把 L3 的编译结果作为节点 backend_lead_graph.add_node("tester", tester_graph.compile()) # ... 用条件边在 coder/tester 之间路由 ... # L1：总控层 root_graph = StateGraph(RootState) root_graph.add_node("backend_lead", backend_lead_graph.compile()) root_graph.add_node("frontend_lead", frontend_lead_graph.compile()) # ...
分层不是越深越好。2-3 层通常够用，再深了调试和状态管理会变得非常复杂。目前来看 2 层（总控 + 执行）对大多数场景已经足够。

4. 平级协作模式

不是所有 Multi-Agent 都需要层级结构。平级协作（Peer Collaboration）模式里，多个 Agent 地位平等，通过「消息传递」或「共享状态」来协作。

  • 方式一：共享 State。多个 Agent 读写同一个 State 的不同字段，通过 State 里的标志位来协调。
  • 方式二：消息传递。Agent A 的输出写入 State 的 messages 列表，Agent B 读取后处理，再把结果 append 进去。本质就是 add_messages 的 reducer 机制。

我更喜欢方式二，因为有显式的通信记录（都在 messages 里），debug 的时候可以完整回溯，而且和 LLM 的对话历史天然对齐。

5. Agent 间通信机制

  • 直接函数调用（Synchronous）：Agent A 的节点函数里直接调用 Agent B 的 invoke。最简单，但耦合度高。
  • 消息队列（Asynchronous）：适合耗时长的任务。LangGraph 本身不提供队列，需要和 Celery/RabbitMQ 等集成。
  • 共享存储（Shared State / Database）：Agent A 写数据库，Agent B 轮询或监听变更。
  • Subgraph 嵌入：把 Agent B 编译后作为 Agent A 图中的一个节点。这是 LangGraph 最推荐的 Agent 组合方式。

这个写法的好处：Reviewer Agent 可以独立测试、独立部署，主图只关心它的输入输出接口。这和微服务架构的思想是一样的。

6. 任务路由与动态分工

在 Multi-Agent 系统里，一个核心问题是：收到一个任务后，派给谁？这就是任务路由（Task Routing）。

  1. LLM 路由（智能路由）：用一个「小型、快速」的 LLM，输入任务描述，输出最优 Agent 名称。灵活，能处理未见过的任务类型，但有延迟和成本。
  2. 规则路由（Deterministic Routing）：用关键词匹配、正则表达式或者向量相似度来路由。快，零 LLM 调用，但只能处理预设的场景。
  3. 混合路由：先走规则路由，规则匹配不上的再走 LLM 路由。推荐这种混合方式，兼顾效率和覆盖率。
我自己的实践经验（虽然还不多）：路由 LLM 不需要用最强的模型，用 GPT-3.5 / GLM-4-Flash 这类又快又便宜的模型做路由完全够用。

7. 状态共享与冲突处理

Multi-Agent 最容易出现 bug 的地方：多个 Agent 同时写 State，出问题了怎么查？

LangGraph 的设计里，State 是中心化的、串行的——图按拓扑序执行，同一时刻只有一个节点在运行，所以不会出现真正的「并发写冲突」。但「逻辑上的冲突」还是会发生：

  • Agent A 写了 state["result"]，Agent B 后面又写了 state["result"]，A 的结果被覆盖了。
  • Agent A 和 B 都往 state["messages"] 里 append，但顺序和预期不符。
  • 条件边的路由依赖于某个标志位，但多个 Agent 都可能修改它，路由行为变得不可预测。

解决方法：用 Reducer 做增量合并、为每个 Agent 分配独立的 State 字段、或用 Subgraph 做状态隔离。

8. 小结

  • Multi-Agent 解决单体 Agent 的 prompt 过长、工具过多、无法并行等问题。
  • Supervisor 模式：中央调度 + 多个 Worker，适合任务可以 centralized 管理的场景。
  • Hierarchical 模式：多层金字塔，每层只关心下一层，适合大型复杂任务。
  • 平级协作模式：共享 State 或消息传递，适合需要频繁交互的场景。
  • Agent 间通信推荐 Subgraph 嵌入方式（编译后当节点），耦合度低、可独立测试。
  • 任务路由：推荐规则 + LLM 混合路由，兼顾效率和覆盖率。
  • 状态共享：用 Reducer 增量合并、为每 Agent 分配独立字段、或用 Subgraph 隔离状态。
第八章 RAG 集成 & 实战项目

Agentic RAG、检索工具设计、端到端案例、调试技巧

1. 为什么 Agent 需要 RAG

学这部分之前，我对 RAG 的理解停留在「检索 + 拼 prompt + 让 LLM 回答」。这套流程在「单轮问答」场景里够用，但放到 Agent 里就有明显局限：

  • 只检索一次。用户的 question 可能有歧义或者需要多轮澄清，一次性检索往往召回不全。
  • 不会反思检索质量。召回的文档不相关？传统 RAG 不会因为「这文档不靠谱」而重新检索。
  • 无法利用工具。用户问「帮我对比 A 和 B 两个文档里的数据」，单次检索只能看到一个视角。

Agentic RAG 的思路是：把检索本身变成一个可由 Agent 调度、反思、重试的工具，而不只是 pipeline 里的一个固定步骤。

RAG 的本质是「给 LLM 提供外部知识」，Agent 的本质是「让 LLM 自主决定下一步做什么」。两者结合后，LLM 可以自主决定「我还需要查更多资料」、「刚才查的不对，换个 query 再查」。

2. Agentic RAG vs Naive RAG

维度 Naive RAG Agentic RAG
检索次数固定 1 次动态，可多轮
Query 生成用户原始问题LLM 重写 / 分解
结果过滤无LLM 评估相关性后决定是否使用
工具使用无可调用 Web Search、SQL 等多种工具
适用场景简单问答研究、分析、多步推理

关键设计是：Agentic RAG 里，检索是一个 Tool，不是 pipeline 的前置步骤。这意味着 Agent 可以在任意时刻调用检索，甚至多次调用不同检索工具（向量库检索 + 网页搜索 + SQL 查询）后综合结果。

3. 检索工具的设计与暴露

@tool def retrieve_docs(query: str) -> list[dict]: """根据查询语句，从向量数据库中检索相关文档片段。 Args: query: 检索查询语句 Returns: 相关文档片段列表，每个元素包含 text 和 metadata """ # 这里是你的向量检索实现（Chroma/Pinecone/Weaviate 等） results = vector_store.similarity_search(query, k=5) return [{"text": doc.page_content, "metadata": doc.metadata} for doc in results] # 把工具绑定到模型 tools = [retrieve_docs, web_search, query_database] llm_with_tools = ChatOpenAI().bind_tools(tools) # 在图里，这个 LLM 可以自主调用 retrieve_docs graph.add_node("model", TalkToModel(llm_with_tools)) graph.add_node("tools", ToolNode(tools)) graph.add_edge("model", "tools") graph.add_conditional_edges("tools", should_continue)

设计检索工具时的几个注意点：

  1. 工具描述（docstring）要写清楚。LLM 根据 docstring 决定「什么时候该调用这个工具」。如果写得太模糊，LLM 会不爱用；写得太窄，又会错过该用的时候。
  2. 返回格式要稳定。返回 list of dict 比返回一段拼接文本好——下游节点可以更灵活地使用每个文档的 metadata。
  3. 考虑返回空结果的情况。检索不到相关文档时，返回空列表 + 一个标志位，让 LLM 有机会决定「要不要换个 query 再查一次」或者「改用网页搜索」。
工具描述是检索工具设计里最值得花时间调的部分。譬如，Harrison 演示过一个例子——改了工具的 docstring 之后，Agent 调用检索工具的准确率明显提升了。

4. 多步检索与自我纠错

这是 Agentic RAG 最有价值的部分：Agent 可以根据第一次检索的结果，决定是否需要进一步检索。

from langgraph.graph import END def should_continue(state: State) -> str: """条件边路由：根据当前状态决定继续检索还是结束。""" messages = state["messages"] last_message = messages[-1] # 如果模型认为已经收集到足够信息，就结束检索 if "FINAL_ANSWER" in last_message.content: return END # 否则，继续调用检索工具 return "retrieve" # 图结构 graph = StateGraph(State) graph.add_node("retrieve", ToolNode([retrieve_docs])) graph.add_node("analyze", analyze_node) # 分析当前检索结果是否足够 graph.add_conditional_edges("analyze", should_continue) graph.set_entry_point("retrieve")

更实用的一种写法是让模型自己判断（而不是靠关键词 FINAL_ANSWER）：

def analyze_node(state: State): # 让模型分析：当前检索结果是否足以回答用户问题？ response = analyzer_llm.invoke([ ("system", "分析当前检索结果是否足以回答用户问题。如果需要更多检索，输出 RETREIVE；如果足够，输出 ANSWER。"), ("user", f"用户问题：{state['question']}\n\n当前检索结果：\n{format_docs(state['retrieved_docs'])}") ]) if "ANSWER" in response.content: return {"next_step": "synthesize_answer"} else: return {"next_step": "retrieve_more"}

这种写法的好处：模型可以基于检索结果的质量动态决定下一步，而不是靠一个固定规则。这比 Naive RAG 的准确率高不少，尤其是在需要多视角信息的复杂问题上。

5. 端到端综合案例

以下是一个综合案例：研究助手 Agent（比如「对比 LangChain 和 LangGraph 在 Agent 构建上的差异」），Agent 自动完成多源检索、信息综合、答案生成。

图的节点设计：

  1. query_rewrite：把用户的自然语言问题，改写成更适合向量检索的 query（可能分解成多个 sub-query）。
  2. retrieve_vector：对向量库做检索，返回 top-K 文档。
  3. retrieve_web：调用 Web Search API，补充向量库里没有的最新信息。
  4. analyze_coverage：分析当前召回的文档是否覆盖了问题的所有方面。如果覆盖不全，回到 query_rewrite 重新检索。
  5. synthesize_answer：综合所有检索结果，生成最终答案，并附上引用来源。
graph = StateGraph(ResearchState) graph.add_node("query_rewrite", query_rewrite_node) graph.add_node("retrieve_vector", vector_retrieve_node) graph.add_node("retrieve_web", web_retrieve_node) graph.add_node("analyze_coverage", analyze_node) graph.add_node("synthesize_answer", synthesize_node) # 并行检索 graph.add_edge("query_rewrite", "retrieve_vector") graph.add_edge("query_rewrite", "retrieve_web") # 两个检索都完成后，进入 analyze graph.add_edge(["retrieve_vector", "retrieve_web"], "analyze_coverage") # 条件边：继续检索 or 生成答案 graph.add_conditional_edges("analyze_coverage", lambda s: "query_rewrite" if s["need_more"] else "synthesize_answer") graph.add_edge("synthesize_answer", END)
这个案例里最巧妙的设计是「并行检索」：向量库和网页搜索可以同时发起，等两个都返回后再做结果融合。LangGraph 的 add_edge(["node_a", "node_b"], "next_node") 语法天然支持这种「多入度节点」的同步等待。

6. 调试技巧与常见坑

  • 坑一：检索工具被频繁调用，token 消耗大。解决：在 State 里加一个 retrieved_cache 字段，同样的 query 不重复检索。
  • 坑二：检索结果太长，撑爆上下文窗口。解决：在检索工具里做摘要（先让小模型把每个文档压缩成 2-3 句），只把摘要传给大模型。
  • 坑三：模型「假装」检索到了内容。解决：在 prompt 里明确告诉模型「如果检索结果为空，必须说『我没有找到相关信息』，禁止编造」。
  • 坑四：多步检索陷入死循环。解决：在 State 里加一个 retrieve_count 计数器，超过 3 次强制进入 synthesize_answer。

调试技巧：

  • 用 stream_mode="messages" 实时看模型在想什么、调了什么工具。
  • 在关键节点里 print(state)（或写日志），记录每步的 State 快照。
  • 用 MemorySaver 做本地调试，每次改动后可以从最后一个 checkpoint 恢复，不用从头跑。
  • 给检索工具加一个「Mock 模式」，返回固定结果，先调通 Agent 的逻辑流，再接真实向量库。

7. 效果评估

RAG Agent 的效果评估，比普通 RAG 更复杂，因为Agent 的行为是不确定的——同样的问题，两次跑可能检索的文档不同、推理路径不同、最终答案也不同。

  • 检索准确率：召回的文档里，有多少是真正相关的。
  • 答案准确率：最终答案是否正确。
  • 检索步数：平均需要几次检索才能给出答案（越少越好，但和准确率需要平衡）。
  • 工具调用正确率：Agent 是否在合适的时机调用了合适的工具。

目前来看，最实用的评估方式是：用一组固定 test case，每次改了 prompt / 工具描述 / 检索参数后，跑一遍所有 case，对比答案质量和检索步数。不需要很复杂的评估框架，暴力但有效。

8. 小结

  • Agentic RAG 把检索变成 Agent 可自主调度、反思、重试的工具，而不只是 pipeline 的前置步骤。
  • 检索工具设计关键：清晰的 docstring、稳定的返回格式、处理空结果的能力。
  • 多步检索：让模型动态判断是否需要进一步检索，用条件边实现循环。
  • 端到端案例：query 重写 → 并行检索（向量 + Web）→ 覆盖分析 → 综合答案。
  • 常见坑：token 消耗大、检索结果太长、模型编造答案、多步检索死循环。
  • 效果评估：检索准确率 + 答案准确率 + 检索步数 + 工具调用正确率。
